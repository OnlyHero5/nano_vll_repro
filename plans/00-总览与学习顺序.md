# plans 总览与学习顺序（按当前仓库重写版）

## 0. 先把这套文档到底在做什么说清楚

`plans/` 不是“当前仓库已经完成到哪”的汇报，也不是“照抄上游代码”的搬运稿。

它真正做的事情只有一件：

> 站在你现在这份仓库的真实状态上，告诉你下一步该怎么改，为什么这样改，改完以后怎么验。

所以你读每一篇时，要同时分清 3 个状态：

1. **当前 HEAD**：也就是你现在仓库里真的有什么、缺什么。
2. **本篇目标状态**：也就是这一篇改完以后，仓库应该变成什么样。
3. **上游参考状态**：也就是 `GeeeekExplorer/nano-vllm` 和 Hugging Face 当前实现给出的“方向”和“语义边界”。

如果这 3 个状态混在一起，文档就会变得很绕，读者也会一边看一边晕。

## 3. 为什么不能直接照搬上游

原因不复杂，直接说人话。

上游和你现在这份仓库，至少有 4 个关键差别：

1. **上游已经是完整工程实现**，你这里还是教学版、单仓库、单进程优先。
2. **上游默认分布式前置条件更强**，你这里很多路径还必须允许单卡直接跑。
3. **你这套文档是教学文档**，所以必须把“为什么”讲出来，不能只给答案。
4. **你当前仓库已经有自己的接口面**，比如 `Sequence`、`Context`、`Scheduler`、测试脚本，不能粗暴覆盖。

所以后面每一篇都必须把下面 3 件事分开写：

1. 上游是怎么设计的。
2. 当前仓库现在是什么状态。
3. 这一篇准备把当前仓库推进到哪里。

## 4. 这次重写后的统一规则

后面所有文档统一按下面这套规则写。

### 4.1 语言规则

1. 先说结论，再说原因。
2. 能用短句说清楚，就不要绕长句。
3. 不写“更抽象”“更优雅”这种空词。
4. 看到容易混淆的地方，直接说“这里最容易误解的是……”。

### 4.2 代码块规则

只允许两种代码块：

1. **完整替换片段**
   这时必须明确写出：
   - 文件名
   - 锚点
   - 替换范围
   - 下面这段代码替换后，文件里应该保留什么不动

2. **完整新增片段**
   这时必须明确写出：
   - 插入位置
   - 插入前后的函数或类
   - 这段代码插进去以后，调用关系会怎么变化

禁止下面这些写法：

- 只贴半段函数体
- 只贴几行 diff，但不说上下文
- 伪代码
- “其余部分类似”“自行补全”

### 4.3 状态标注规则

每一篇都要把这 3 句话写清楚：

1. **当前仓库现在是什么状态**
2. **本篇改完以后要变成什么状态**
3. **本篇默认依赖前面哪些篇章已经落地**

这样读者才不会把“目标状态”错看成“当前状态”。

## 7. 开始每一篇之前的固定动作

### 7.1 先看当前文件，不要先抄答案

至少确认下面 4 件事：

1. 当前类名和函数名是不是已经存在。
2. 当前测试到底调用了什么接口。
3. 当前注释里哪些是对的，哪些已经过期。
4. 这个文件有没有被别的模块强依赖。

### 7.2 先跑最小验收，再进入下一篇

当前仓库最现实的入口仍然是：

```bash
cd nano_vll_repro
python tests/test_Day1.py
python tests/test_Day2.py
python tests/test_Day3.py
python tests/test_Day4.py
python example.py
```

如果你环境里装了 `pytest`，再补一层：

```bash
python -m pytest tests -q
```

但要记住：

- 当前仓库很多测试依赖 `transformers`
- `layers/__init__.py` 会导入 `Attention`
- 所以如果没装 `flash_attn`，`tests/test_Day4.py` 可能在导入阶段就失败

这不是文档问题，而是环境前提问题。

### 7.3 不要跳篇改

最容易出事的 3 个跳篇动作是：

1. `03` 还没做完，就先把 `ModelRunner` 改成 `top_k / top_p` 全量依赖。
2. `04` 还没收口，就先做 TP。
3. `run_model()`、`compute_logits()` 的边界还没建好，就先做 CUDA Graph。
