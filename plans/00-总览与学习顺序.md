## 0. 先说清楚这一版文档和旧版的区别

旧版 `plans/` 最大的问题不是“有些实现细节不够严谨”，而是写法本身就不适合教学：

1. 大量使用“把整个文件替换成下面这份代码”的表达，这会掩盖你当前仓库到底已经写到哪一步、缺口具体在哪里。
2. 代码块里的注释密度太低，很多地方只有结论，没有把“为什么这样改”讲透。
3. 没有把“上游 `nano-vllm` 可以直接照搬的部分”和“你当前仓库必须按现状改写的部分”分开说明。
4. 对测试入口、接口漂移、单卡与多卡路径的先后顺序交代不够，容易导致你越改越乱。

这次重写后的规则只有一条：

> 后续每一篇都只指导你“基于当前仓库逐段修改”，不会再把“整文件替换”当成默认方案。

## 2. 为什么不能直接照搬上游 `nano-vllm`

虽然上游是主要参考，但你的仓库和上游并不等价，至少有 5 个重要差异：

1. 你的仓库是“单仓库、单进程优先”的教学实现，而上游已经有更完整的多进程 worker 结构。
2. 你的 `Context`、`Scheduler`、`LLMEngine`、测试脚本都已经形成了自己的 API 面，不能粗暴套进去。
3. 你的代码里有大量中文注释，教学目标比“最短可运行代码”更强，因此注释标准必须高于上游。
4. 上游默认很多地方假设 `dist.init_process_group()` 已完成，而你当前 Day1~Day5 的大量路径必须允许“未初始化分布式时仍可单卡运行”。
5. 你的 `plans/` 目标不是“抄出一份工作代码”，而是“指导你继续自己往下写”。这两者的文档写法完全不同。

所以后面每一篇都会明确区分：

- 哪些是“上游设计意图”
- 哪些是“你当前仓库的局部修改点”
- 哪些是“为了和后续 Day6 / Day7 衔接而提前埋的接口”

## 3. 这套教案的统一写法

从这一篇开始，后续 `01 ~ 07` 都按同一模板来写：

1. 本篇目标：本篇做完后，仓库应该新增什么能力。
2. 权威参考：本篇到底对照哪几个上游文件，不再笼统写“参考 nano-vllm”。
3. 先看当前仓库：先指出你当前文件里已经有什么、断在哪里。
4. 修改原则：告诉你这篇不应该整文件推翻，而应该保留哪些已有结构。
5. 逐步修改：每一步都说明“改哪一段、为什么改、改完影响什么”。
6. 带详细注释的代码片段：只给局部补丁级别的代码，并保证注释密度明显高于旧版。
7. 验收命令：先做最小验收，再跑阶段测试。
8. 常见错误：把最容易踩坑的接口漂移写出来。

你可以把后面的代码块理解成：

> “这是你应该手动改到当前文件里的目标片段”，不是“允许你无脑覆盖整个文件”的最终答案。

---

## 4. 代码注释标准

你这次对文档的要求很明确，所以这里把标准写死：

1. 文档里的指导代码块，必须出现类/函数输入输出说明。
2. 复杂分支必须写行间注释，解释“为什么不是另一种写法”。
3. 教学片段里的注释密度，目标是不低于总代码行数的 50%。
4. 对“兼容当前仓库”和“对齐上游实现”的桥接逻辑，必须解释。

尤其注意：

- “这一行在做什么”不够。
- 必须解释“这一行为什么要在这里做，而不是放到别的层”。

## 7. 进入每一篇之前要做的固定动作

### 7.1 先读当前文件，不要先读“答案”

每一篇开始前，先把对应文件打开，至少确认：

- 当前类和函数名是否已经存在
- 当前测试脚本到底调用了什么接口
- 当前代码里哪些注释可以保留
- 哪些字段已经被别的模块依赖

### 7.2 先做最小验收，再进入下一篇

固定命令建议：

```bash
cd nano_vll_repro
python -m py_compile config.py sampling_params.py llm.py
python -m pytest tests -q
```

如果你这一篇只是局部推进，不要求一次跑完整仓库，也至少要跑：

- 本篇涉及的单文件语法检查
- 本篇涉及的 `tests/test_DayX.py`

### 7.3 不要跳篇改

这套教案是强顺序依赖的，尤其是：

- `03` 之前，不要把所有调用点都强行升级成四参数 sampler
- `05` 之前，不要在模型里半做 TP 头数切分
- `06` 之前，不要先写图捕获，再回头补 decode 静态输入
