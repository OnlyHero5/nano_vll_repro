## 3. 先看当前仓库的真实缺口

### 3.1 `engine/model_runner.py`

当前 [engine/model_runner.py](/home/psx/nano_vllm_repro/nano_vll_repro/engine/model_runner.py:319) 至少有 6 个真实问题：

1. 仍然假设 `self.model(input_ids, positions)` 直接返回 logits，但你前一篇应该已经把模型收口成“forward 返回 hidden states”。
2. 采样时只会临时构造 `temperatures`，还拿不到 `top_k / top_p`。
3. `_load_model()` 里把模型 dtype 写死成 `torch.bfloat16`，和 `Config.torch_dtype` 脱节。
4. `allocate_kv_cache()` 里把 cache dtype 写死成 `torch.float16`，和 `Config.kv_torch_dtype` 脱节。
5. 每次 `run()` 结束后没有统一 `reset_context()`，后面做 graph 时会埋隐蔽 bug。
6. `prepare_prefill()` / `prepare_decode()` 已经可用，但采样元数据没有独立抽象出来。

### 3.2 `engine/scheduler.py`

当前 [engine/scheduler.py](/home/psx/nano_vllm_repro/nano_vll_repro/engine/scheduler.py:163) 的问题不在“大框架完全不存在”，而在账本状态维护过粗：

1. `postprocess()` append 完 token 后，没有把“本轮真正进入模型的旧 token 数”记到 `num_cached_tokens`。

## 4. 本篇修改原则

### 4.1 先收口运行时边界，再谈更复杂调度

这一篇先不追上游的：

- chunked prefill
- partial prefill
- worker 多进程
- prefix-cache 驱动的再 prefill

当前目标非常明确：

> 先让你当前这份仓库的“单卡 generate 主循环”接口统一、状态清晰、测试可验收。

### 4.2 采样元数据必须始终从 `Sequence` 读取

这一步要把一个设计原则写死：

1. `LLMEngine.add_request()` 把用户输入封装成 `Sequence`
2. 从此以后，运行时只读 `Sequence`
3. 不允许 `ModelRunner` 再回头依赖外部原始 `SamplingParams`

### 4.3 现在就为 Day6 留出 `run_model()` 边界

后面做 CUDA Graph 时，最自然的切入点就是：

1. `run()` 负责准备输入、调用 `run_model()`、调用 sampler
2. `run_model()` 负责“主干 hidden states -> logits”

所以本篇必须先把这层边界搭起来。

## 5.2 新增 `prepare_sampling_tensors()`

修改位置：

- 文件：`nano_vll_repro/engine/model_runner.py`
- 操作：在 `prepare_decode()` 后、`run_model()` / `run()` 前插入下面这份完整方法

完整新增代码如下：

```python
def prepare_sampling_tensors(
    self,
    sequences: list[Sequence],
) -> tuple[torch.Tensor, torch.Tensor, torch.Tensor]:
    """
    从当前 step 的 Sequence 列表中提取采样参数张量。

    输入：
    - 当前 step 要执行的序列列表

    输出：
    1. temperatures: `[batch]`
    2. top_ks: `[batch]`
    3. top_ps: `[batch]`

    之所以单独拆函数，而不是在 `run()` 里临时写三段 tensor 构造，
    是因为这些张量和 input_ids / positions 属于两种不同职责：
    - 前者是采样元数据
    - 后者是模型前向输入
    """

    # 每条 Sequence 的采样配置已经在 `Sequence.__init__()` 里固化成运行期状态。
    # 这里不再回头读取外部 `SamplingParams`，只认当前请求自己的状态。
    temperatures = torch.tensor(
        [seq.temperature for seq in sequences],
        dtype=torch.float32,
        device=self.device,
    )
    top_ks = torch.tensor(
        [seq.top_k for seq in sequences],
        dtype=torch.long,
        device=self.device,
    )
    top_ps = torch.tensor(
        [seq.top_p for seq in sequences],
        dtype=torch.float32,
        device=self.device,
    )

    return temperatures, top_ks, top_ps
```

这里最容易被低估的收益是：

1. 以后如果你要加 `repetition_penalty`
2. 或者要加 per-sequence `beam_size`
3. 都可以继续沿这个模式往下扩

## 5.3 新增 `run_model()`，明确“主干前向 + logits 计算”边界

修改位置：

- 文件：`nano_vll_repro/engine/model_runner.py`
- 操作：在 `prepare_sampling_tensors()` 后、`run()` 前插入下面这份完整方法

完整新增代码如下：

```python
@torch.inference_mode()
def run_model(
    self,
    input_ids: torch.Tensor,
    positions: torch.Tensor,
) -> torch.Tensor:
    """
    运行模型主干，并把 hidden states 转成 logits。

    输入：
    1. input_ids: 当前 step 的 token 输入
    2. positions: 当前 step 的位置索引

    输出：
    - vocab 维 logits

    当前阶段故意把这层边界显式抽出来，原因是：
    1. 单卡路径下，调用方最终只关心 logits
    2. 但内部已经为 Day6 留出了“主干 replay、logits 仍在 graph 外算”的空间
    """

    # 前一篇已经把模型接口收口成：
    # - forward 返回 hidden states
    # - compute_logits 单独负责 lm_head
    hidden_states = self.model(input_ids, positions)
    logits = self.model.compute_logits(hidden_states)
    return logits
```

这里一定不要再继续写成：

```python
logits = self.model(input_ids, positions)
```

因为那会把上一篇刚收口好的模型边界又重新抹平。

```python
def _load_model(self) -> nn.Module:
    """
    加载模型结构与权重，并迁移到当前 device。

    这里的职责划分必须清楚：
    1. `Qwen3ForCausalLM.from_pretrained()` 只负责创建结构
    2. `load_model(...)` 才负责把 safetensors 权重写进去
    3. dtype 选择统一走 `Config.torch_dtype`
    """

    from models.qwen3 import Qwen3ForCausalLM

    print(f"[ModelRunner] 加载模型：{self.config.model_path}")

    # 先创建结构。
    model = Qwen3ForCausalLM.from_pretrained(self.config.model_path)

    # 再把本地权重写入。
    load_model(model, self.config.model_path)

    # dtype 统一从 Config 取，不要在这里写死 BF16。
    model = model.to(self.device, dtype=self.config.torch_dtype)
    model.eval()

    print(f"[ModelRunner] 模型加载完成，设备：{self.device}")
    return model
```

### `allocate_kv_cache()` 完整替代代码

```python
def allocate_kv_cache(self, num_blocks: int) -> None:
    """
    预分配 KV Cache。

    当前单卡版本每层 cache 的形状仍然是：
    `[2, num_blocks, block_size, num_kv_heads, head_dim]`

    这里故意把 dtype 统一交给 `Config.kv_torch_dtype`，
    原因是后面会出现：
    - 主干权重用 BF16
    - KV Cache 仍然用 FP16

    这两种 dtype 语义不应该在这里被写死。
    """

    bytes_per_block = (
        2
        * self.block_size
        * self.num_kv_heads
        * self.head_dim
        * torch.tensor([], dtype=self.config.kv_torch_dtype).element_size()
    )
    total_bytes = self.num_layers * num_blocks * bytes_per_block
    print(f"[ModelRunner] KV Cache 显存需求：{total_bytes / 1024**3:.2f} GB")

    self.kv_cache = []
    for _ in range(self.num_layers):
        cache = torch.zeros(
            2,
            num_blocks,
            self.block_size,
            self.num_kv_heads,
            self.head_dim,
            dtype=self.config.kv_torch_dtype,
            device=self.device,
        )
        self.kv_cache.append(cache)

    print(f"[ModelRunner] KV Cache 分配完成：{num_blocks} 块 × {self.num_layers} 层")
```

这里真正重要的不是“更优雅”，而是：

1. dtype 决策应该统一在 `Config`
2. 运行时模块只消费已经整理好的配置

## 5.5 重写 `run()`：统一准备输入、跑模型、采样，并在末尾 reset Context

修改位置：

- 文件：`nano_vll_repro/engine/model_runner.py`
- 操作：把 `run()` 整个方法替换为下面这份完整实现

完整替代代码如下：

```python
@torch.inference_mode()
def run(
    self,
    sequences: list[Sequence],
    is_prefill: bool,
) -> list[int]:
    """
    执行一个 step 的模型推理与采样。

    输入：
    1. sequences: 当前 step 要执行的序列列表
    2. is_prefill: 当前是否为 prefill 阶段

    输出：
    - 当前 step 每条序列对应的下一个 token id 列表

    执行顺序必须固定：
    1. 根据阶段准备输入与 Context
    2. 构造采样元数据张量
    3. 调用 `run_model()` 获得 logits
    4. prefill 路径只保留每条序列最后一个 token 的 logits
    5. 调 sampler
    6. 在函数末尾统一 reset Context
    """

    if not sequences:
        return []

    # Prefill 和 Decode 的输入结构不同，但两条路径最终都会汇合到统一的 `run_model()`。
    if is_prefill:
        input_ids, positions = self.prepare_prefill(sequences)
    else:
        input_ids, positions = self.prepare_decode(sequences)

    # 采样参数统一从 Sequence 中提取。
    temperatures, top_ks, top_ps = self.prepare_sampling_tensors(sequences)

    # 统一走已经收口好的运行时边界。
    logits = self.run_model(input_ids, positions)

    if is_prefill:
        # Prefill 会一次性前向所有 prompt token。
        # 但真正用于“采样下一个 token”的，只有每条序列最后一个 prompt token 的 logits。
        context = get_context()
        last_token_indices = context.cu_seqlens_q[1:] - 1
        logits = logits[last_token_indices.long()]

    next_tokens = self.sampler(logits, temperatures, top_ks, top_ps)

    # 这条 reset 纪律一定不能省。
    # 原因是 Context 是“当前 step 的元数据快照”，不是跨 step 的全局状态。
    reset_context()
    return next_tokens.tolist()
```

这一步要特别注意：

1. 不要给 prefill / decode 各写一套采样逻辑
2. 不要把 `reset_context()` 放到更深层的方法里

统一外层 reset，后面做 graph 时最稳。

## 5.7 重写 `LLMEngine.step()`，改掉 prefill token 统计

修改位置：

- 文件：`nano_vll_repro/engine/llm_engine.py`
- 操作：把 `step()` 整个方法替换为下面这份完整实现

完整替代代码如下：

```python
def step(self) -> tuple[list[tuple[int, list[int]]], int]:
    """
    执行单步推理。

    输出：
    1. outputs: 本轮新完成序列的 `(seq_id, completion_token_ids)` 列表
    2. num_tokens:
       - prefill 阶段返回本轮真正进入模型的新 token 数
       - decode 阶段返回负的序列数，便于上层区分 throughput 统计口径
    """

    seqs, is_prefill = self.scheduler.schedule()
    if not seqs:
        return [], 0

    # 注意：prefill token 统计必须放在 postprocess 之前。
    # 否则 append 之后的长度会把“刚生成但尚未进入模型的新 token”也算进去。
    if is_prefill:
        num_tokens = sum(max(seq.num_tokens - seq.num_cached_tokens, 1) for seq in seqs)
    else:
        num_tokens = -len(seqs)

    token_ids = self.model_runner.run(seqs, is_prefill)
    finished_seqs = self.scheduler.postprocess(seqs, token_ids)

    outputs = [
        (seq.seq_id, seq.completion_token_ids)
        for seq in finished_seqs
    ]
    return outputs, num_tokens
```

这里最重要的不是那一行求和公式本身，而是：

1. prefill throughput 必须统计“本轮真正进入模型的新 token”
2. 不能继续拿 `len(seq)` 这种混着历史 token 的口径

```python
    # 沿用 Qwen chat template，保持和实际聊天用法一致。

        tokenizer.apply_chat_template(
            [{"role": "user", "content": prompt}],
            tokenize=False,
            add_generation_prompt=True,
        )
        for prompt in raw_prompts
    ]

    # 这里故意显式传入采样参数，而不是依赖默认值，
    # 这样读者能直接看到 `LLM.generate()` 当前接受的采样控制项。
    sampling_params = SamplingParams(
        temperature=0.8,
        top_k=20,
        top_p=0.9,
        max_tokens=64,
    )

    outputs = llm.generate(
        prompts,
        sampling_params=sampling_params,
        use_tqdm=False,
    )

    # `LLM.generate()` 当前返回的是 `list[dict]`，
    # 每个元素至少有 `text` 和 `token_ids` 两个字段。
    for prompt, output in zip(raw_prompts, outputs):
        print(f"\n[问题] {prompt}")
        print(f"[回答] {output['text']}")
        print(f"[生成 token 数] {len(output['token_ids'])}")


if __name__ == "__main__":
    if torch.cuda.is_available():
        print(f"CUDA: {torch.cuda.get_device_name(0)}")
    main()
```

```python
"""Day 5 测试脚本 - 单卡推理主链路验收

这份文件只验证 Day5 收口后的 3 件事情：
1. ModelRunner 是否能把每条 Sequence 的采样参数提取成张量
2. Scheduler.postprocess 是否正确维护 num_cached_tokens
3. LLM.generate 是否能在本地模型存在时走通最小单卡烟雾链路

注意：
1. 前两个测试可以在 CPU 上执行
2. 第三个 smoke test 依赖本地模型权重与 CUDA，环境不满足时允许显式跳过
"""

import os
import sys
from types import SimpleNamespace

import torch


PROJECT_ROOT = os.path.join(os.path.dirname(__file__), "..")
sys.path.insert(0, PROJECT_ROOT)


from engine.block_manager import BlockManager
from engine.model_runner import ModelRunner
from engine.scheduler import Scheduler
from engine.sequence import Sequence, SequenceStatus
from llm import LLM
from sampling_params import SamplingParams


@torch.inference_mode()
def test_prepare_sampling_tensors() -> None:
    """验证 ModelRunner.prepare_sampling_tensors 的输出。"""

    # 这个测试只关心“采样参数抽取”，不值得为它加载整套模型。
    # 因此直接用 __new__ 绕过重量级初始化，只补一个 device 就够了。
    runner = ModelRunner.__new__(ModelRunner)
    runner.device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

    seq1 = Sequence(
        [1, 2, 3],
        SamplingParams(
            temperature=0.0,
            top_k=0,
            top_p=1.0,
            max_tokens=8,
        ),
    )
    seq2 = Sequence(
        [4, 5, 6],
        SamplingParams(
            temperature=0.8,
            top_k=20,
            top_p=0.9,
            max_tokens=16,
        ),
    )

    temperatures, top_ks, top_ps = ModelRunner.prepare_sampling_tensors(runner, [seq1, seq2])

    # 先锁住形状。
    assert temperatures.shape == (2,)
    assert top_ks.shape == (2,)
    assert top_ps.shape == (2,)

    # 再锁住数值，确保不同 Sequence 的配置没有串位。
    assert temperatures.tolist() == [0.0, 0.8]
    assert top_ks.tolist() == [0, 20]
    assert torch.allclose(top_ps.cpu(), torch.tensor([1.0, 0.9]))

    print("✅ prepare_sampling_tensors 测试通过")


@torch.inference_mode()
def test_scheduler_postprocess_updates_cached_tokens() -> None:
    """验证 Scheduler.postprocess 是否正确维护 cached-token 账本。"""

    # postprocess 只依赖少数字段，不需要完整 Config。
    config = SimpleNamespace(
        max_num_seqs=4,
        max_num_batched_tokens=64,
        eos=99999,
    )

    block_manager = BlockManager(num_blocks=8, block_size=Sequence.block_size)
    scheduler = Scheduler(config, block_manager)

    seq = Sequence(
        [10, 11, 12],
        SamplingParams(
            temperature=0.7,
            top_k=10,
            top_p=0.9,
            max_tokens=4,
        ),
    )

    # 模拟进入 running 队列，并且已经拿到了 block。
    seq.status = SequenceStatus.RUNNING
    block_manager.allocate(seq)
    scheduler.running.append(seq)

    old_num_tokens = seq.num_tokens
    finished = scheduler.postprocess([seq], [13])

    # 当前这一步不应结束序列。
    assert finished == []
    assert seq.status == SequenceStatus.RUNNING

    # 新 token 已 append，因此总长度加一。
    assert seq.num_tokens == old_num_tokens + 1

    # cached token 数必须等于 append 前的旧长度。
    assert seq.num_cached_tokens == old_num_tokens

    print("✅ scheduler.postprocess 账本测试通过")


@torch.inference_mode()
def test_llm_generate_smoke() -> None:
    """验证最小单卡 generate 链路。"""

    model_path = os.path.join(PROJECT_ROOT, "models", "Qwen3-0.6B")

    # 当前教学仓库的完整推理路径依赖本地权重和 CUDA。
    if not torch.cuda.is_available():
        print("skip: 单卡 smoke test 需要 CUDA")
        return

    if not os.path.isdir(model_path):
        print(f"skip: 模型路径不存在 {model_path}")
        return

    llm = LLM(model_path)
    sampling_params = SamplingParams(
        temperature=0.7,
        top_k=20,
        top_p=0.9,
        max_tokens=8,
    )

    prompts = ["你好，请用一句话解释一下 PagedAttention。"]
    outputs = llm.generate(prompts, sampling_params=sampling_params, use_tqdm=False)

    # 先锁住返回结构。
    assert isinstance(outputs, list)
    assert len(outputs) == 1
    assert "text" in outputs[0]
    assert "token_ids" in outputs[0]

    # 再锁住最小有效性。
    assert isinstance(outputs[0]["text"], str)
    assert isinstance(outputs[0]["token_ids"], list)
    assert len(outputs[0]["token_ids"]) > 0

    print("✅ LLM.generate 单卡 smoke test 通过")


if __name__ == "__main__":
    print("=" * 60)
    print("Day 5 单卡链路测试开始")
    print("=" * 60)

    test_prepare_sampling_tensors()
    test_scheduler_postprocess_updates_cached_tokens()
    test_llm_generate_smoke()

    print("=" * 60)
    print("🎉 Day 5 测试执行完成")
    print("=" * 60)
```

## 7. 常见错误

### 7.1 `run()` 里忘记调用 `compute_logits()`

后果：

- 采样器拿到的是 hidden states，不是 vocab logits
- 报错往往会在 softmax 阶段才出现，更难定位

### 7.2 `reset_context()` 忘记放在 `run()` 末尾

后果：

- 当前单卡路径偶尔还能侥幸通过
- 到 Day6 做 graph replay 时会非常难查

### 7.3 `num_cached_tokens` 在 append 之后才更新

后果：

- 会把“刚生成但尚未进入模型的新 token”也记成 cached
- decode 上下文长度会慢慢错位

### 7.4 `LLMEngine.step()` 继续拿 `len(seq)` 统计 prefill token

后果：

- 进度条里的 throughput 会严重失真
- Day7 benchmark 时你会得到好看但没意义的数据

## 8. 本篇真正学到的东西

这一篇最关键的不是“终于能 generate 了”，而是下面 4 件事：

1. 单卡主循环也必须尽早把边界拆干净。
2. 采样参数应该跟着 `Sequence` 走，而不是跟着外部入参走。
3. `Context` 清理必须现在就形成纪律，而不是等 Day6 再补。
4. 统计口径如果不先理清，后面的 benchmark 基本没有可信度。
