# 04. 补齐单卡推理链路与 Day5 测试

## 1. 本篇目标

这一篇只做一件事：

> 把“当前仓库的单卡 generate 主循环”收口成一条真正能继续演进的链路。

这里的“收口”不是重写整个推理框架，而是把下面 4 个边界讲清楚并落到代码里：

1. `ModelRunner` 负责准备输入、跑模型、准备采样张量。
2. `Scheduler` 负责维护序列状态和 KV Cache 账本。
3. `LLMEngine` 负责串起一次完整 step。
4. `example.py` 和 `tests/test_Day5.py` 负责把这条链路锁住。

本篇完成后，你至少应该得到下面 5 个结果：

1. `ModelRunner` 不再把“主干前向”和“采样”糊在一起。
2. 采样张量真正从 `Sequence` 读取，而不是临时手搓温度。
3. `Scheduler.postprocess()` 会维护最基础的 `num_cached_tokens`。
4. `LLMEngine.step()` 的 token 统计不再混着历史 token。
5. 仓库里新增一个真正对应单卡主链路的 `tests/test_Day5.py`。

## 3. 先看当前仓库的真实状态

### 3.1 `engine/model_runner.py`

当前 [engine/model_runner.py](/home/psx/nano_vllm_repro/nano_vll_repro/engine/model_runner.py:319) 的核心问题有 4 个：

1. `run()` 里直接 `self.model(input_ids, positions)`，当前逻辑默认模型直接回 logits。
2. 采样时只构造了 `temperatures`，没有 `top_k / top_p`。
3. `run()` 同时管准备输入、跑模型、取最后位置、采样，职责太挤。
4. `run()` 结束后没有统一 `reset_context()`，后面做 CUDA Graph 会很危险。

### 3.2 `engine/scheduler.py`

当前 [engine/scheduler.py](/home/psx/nano_vllm_repro/nano_vll_repro/engine/scheduler.py:149) 的问题不在主框架，而在账本太粗：

1. `postprocess()` 只 append 新 token。
2. 但它没有把“这一轮哪些 token 已经进入 cache”固化回 `num_cached_tokens`。

结果就是：

- Prefix Cache 账本不稳
- preemption 之后状态更难解释
- 后面统计 prefill/decode token 数时容易混乱

2. `step()` 没把“本轮新计算 token 数”和“本轮完成了哪些序列”分得很清楚。

## 4. 本篇修改原则

### 4.1 先收口边界，不追高级特性

这一篇不追：

- TP
- CUDA Graph
- chunked prefill
- worker 多进程

这篇只追一个目标：

> 单卡主循环边界干净，状态可解释，后面能继续接着加功能。

### 4.2 采样参数一律从 `Sequence` 读

原因很简单。

一条请求一旦进了系统，后面的运行时只应该认这条 `Sequence` 当前的状态。

不要让 `ModelRunner` 再回头找外部原始 `SamplingParams`。

### 4.3 `run_model()` 要先于 CUDA Graph 出现

因为后面做 graph 时，最自然的切点就是：

1. `prepare_prefill() / prepare_decode()`
2. `run_model()`
3. `sampler(...)`

如果现在不先把这层边界搭起来，后面 graph 会直接和采样逻辑缠死。

## 5. 逐步修改

## 5.1 在 `ModelRunner` 里新增 `prepare_sampling_tensors()`

修改位置：

- 文件：`engine/model_runner.py`
- 锚点：插在 `prepare_decode()` 后面，`run()` 前面

完整新增代码如下：

```python
def prepare_sampling_tensors(
    self,
    sequences: list[Sequence],
) -> tuple[torch.Tensor, torch.Tensor, torch.Tensor]:
    """
    从当前 step 的序列列表中提取采样参数张量。

    返回：
    1. temperatures: [batch]
    2. top_ks: [batch]
    3. top_ps: [batch]
    """
    temperatures = torch.tensor(
        [seq.temperature for seq in sequences],
        dtype=torch.float32,
        device=self.device,
    )
    top_ks = torch.tensor(
        [seq.top_k for seq in sequences],
        dtype=torch.long,
        device=self.device,
    )
    top_ps = torch.tensor(
        [seq.top_p for seq in sequences],
        dtype=torch.float32,
        device=self.device,
    )
    return temperatures, top_ks, top_ps
```

这一步最重要的不是省几行代码，而是把职责拆开：

- `prepare_prefill / prepare_decode` 管模型输入
- `prepare_sampling_tensors` 管采样元数据

## 5.2 在 `ModelRunner` 里新增 `run_model()`

修改位置：

- 文件：`engine/model_runner.py`
- 锚点：插在 `prepare_sampling_tensors()` 后面

完整新增代码如下：

```python
@torch.inference_mode()
def run_model(
    self,
    input_ids: torch.Tensor,
    positions: torch.Tensor,
    is_prefill: bool,
) -> torch.Tensor:
    """
    执行模型主干并返回用于采样的 logits。

    前置条件：
    - `Qwen3ForCausalLM.forward()` 返回 hidden states
    - `Qwen3ForCausalLM.compute_logits()` 单独负责 lm head 投影
    """
    hidden_states = self.model(input_ids, positions)
    logits = self.model.compute_logits(hidden_states)

    # Prefill 一次会喂入整段 prompt，因此只取每条序列最后一个位置的 logits。
    if is_prefill:
        context = get_context()
        last_token_indices = context.cu_seqlens_q[1:] - 1
        logits = logits[last_token_indices.long()]

    return logits
```

这一步的核心收益是：

> 从这里开始，`run()` 不再需要知道“模型内部到底怎么从 hidden states 变成 logits”。

## 5.3 用下面这份完整 `run()` 替换当前 `ModelRunner.run()`

修改位置：

- 文件：`engine/model_runner.py`
- 锚点：`@torch.inference_mode() def run(...)`

完整替代代码如下：

```python
@torch.inference_mode()
def run(
    self,
    sequences: list[Sequence],
    is_prefill: bool,
) -> list[int]:
    """
    执行当前 step 的单卡推理，并返回每条序列的下一个 token。
    """
    if not sequences:
        return []

    if is_prefill:
        input_ids, positions = self.prepare_prefill(sequences)
    else:
        input_ids, positions = self.prepare_decode(sequences)

    temperatures, top_ks, top_ps = self.prepare_sampling_tensors(sequences)

    try:
        logits = self.run_model(input_ids, positions, is_prefill)
        next_tokens = self.sampler(logits, temperatures, top_ks, top_ps)
        return next_tokens.tolist()
    finally:
        # Context 是“当前 step 的元数据快照”。
        # 无论本轮是 eager 成功、采样报错，还是后面 graph replay 失败，
        # 都必须在 step 结束时统一清掉，防止旧 batch 数据漏到下一轮。
        reset_context()
```

这份代码和当前版本最大的不同有两点：

1. 采样参数不再只剩温度。
2. `reset_context()` 被提升成强纪律，而不是“后面再说”。

## 5.4 用下面这份完整 `postprocess()` 替换当前 `Scheduler.postprocess()`

修改位置：

- 文件：`engine/scheduler.py`
- 锚点：`def postprocess(...)`

完整替代代码如下：

```python
def postprocess(
    self,
    seqs: List[Sequence],
    token_ids: List[int],
    is_prefill: bool,
) -> List[Sequence]:
    """
    更新当前 step 结束后的序列状态，并返回本轮完成的序列。
    """
    finished_seqs: List[Sequence] = []

    for seq, token_id in zip(seqs, token_ids):
        # 先把这轮真正进入模型的旧 token 账本补齐。
        #
        # Prefill 阶段：当前 prompt 已经真正写入 cache，
        # 因此此时 num_cached_tokens 至少应该追到当前旧 token 数。
        #
        # Decode 阶段：新 token 在本轮前向里已经写入 cache，
        # 但 append_token 还没执行，所以写回的仍是“旧长度”。
        seq.num_cached_tokens = max(seq.num_cached_tokens, seq.num_tokens)

        # 再把新生成 token 接到序列尾部。
        seq.append_token(token_id)

        is_eos = (not seq.ignore_eos) and (token_id == self.eos)
        is_max_tokens = seq.num_completion_tokens >= seq.max_tokens

        if is_eos or is_max_tokens:
            seq.status = SequenceStatus.FINISHED
            self.block_manager.deallocate(seq)
            self.running.remove(seq)
            finished_seqs.append(seq)

    return finished_seqs
```

注意这里我没有把 `is_prefill` 写成条件分支去分别更新。

原因是当前这两个阶段都满足同一条账本规则：

> 本轮进入模型的旧 token，在 step 结束时就应该被视为已经缓存。

## 5.5 用下面这份完整 `step()` 替换当前 `LLMEngine.step()`

修改位置：

- 文件：`engine/llm_engine.py`
- 锚点：`def step(...)`

完整替代代码如下：

```python
def step(self) -> tuple[list[tuple[int, list[int]]], int]:
    """
    执行单步推理。

    返回：
    1. outputs: 本轮刚刚完成的序列输出
    2. num_tokens: 本轮真正执行的 token 数
       - Prefill: 正数，表示本轮真正新算的 prompt token 数
       - Decode: 负数，绝对值等于本轮 decode 的序列数
    """
    seqs, is_prefill = self.scheduler.schedule()
    if not seqs:
        return [], 0

    if is_prefill:
        num_tokens = sum(len(seq) - seq.num_cached_tokens for seq in seqs)
    else:
        num_tokens = -len(seqs)

    token_ids = self.model_runner.run(seqs, is_prefill)
    finished_seqs = self.scheduler.postprocess(seqs, token_ids, is_prefill)

    outputs = [
        (seq.seq_id, seq.completion_token_ids)
        for seq in finished_seqs
    ]
    return outputs, num_tokens
```

这一步最大的变化只有一句话：

> prefill 统计应该统计“本轮新算了多少 token”，而不是“这些序列现在总共有多少 token”。

## 7. 常见错误

### 7.1 把 `run_model()` 写成又做采样又做模型前向

这样后面做 CUDA Graph 会非常难拆。

### 7.2 继续在 `run()` 里临时手搓温度，不从 `Sequence` 读

这样 `top_k / top_p` 很快又会丢。

### 7.3 prefill token 统计继续用 `len(seq)`

这会把历史 token 算进去，吞吐量统计会失真。

### 7.4 `postprocess()` 只 append token，不更新 `num_cached_tokens`

后面 Prefix Cache 和 preemption 语义都会变糊。

### 7.5 `reset_context()` 还是放到“以后 graph 再说”

这类问题现在不修，后面只会更难查。

## 8. 本篇真正学到的东西

单卡主循环真正重要的不是“把结果跑出来”。

而是把下面这 4 层边界分开：

1. 调度谁
2. 准备什么输入
3. 模型算出什么
4. 采样和后处理怎么收尾

边界一旦干净，后面的 TP 和 CUDA Graph 才不是硬堆功能。
