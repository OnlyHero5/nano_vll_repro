## 3. 先看当前仓库的真实缺口

### 3.1 `engine/model_runner.py`

当前文件至少有 5 个问题：

1. 仍然假设 `self.model(input_ids, positions)` 直接返回 logits，和 `02` 的接口收口不一致。
2. 采样时只会构造 `temperatures`，拿不到 `top_k / top_p`。
3. 模型 dtype 写死成 `torch.bfloat16`，和 `Config.torch_dtype` 脱节。
4. 没有在每次 run 结束后 `reset_context()`，后面做 CUDA Graph 时会出隐蔽 bug。
5. `prepare_decode()` 和 `prepare_prefill()` 是可用的，但采样元数据没有独立抽出来。

### 3.2 `engine/scheduler.py`

当前问题不在“调度大框架”，而在状态维护太粗：

1. `postprocess()` append 完 token 后，没有更新 `num_cached_tokens`
2. 这样后面无论是性能统计还是重入 decode，都缺少最基本的账本

### 3.3 `engine/llm_engine.py`

当前文件已经能跑，但还有 3 个不利于后续演进的点：

1. `step()` 的 prefill token 统计不准确
2. `generate()` 没有真正利用 `SamplingParams` 的新字段
3. 初始化和打印逻辑有些分散，后面做 TP 时需要更干净的边界

### 3.4 `example.py`

当前示例能当烟雾测试，但不够像“下一步要继续开发的入口”：

- 没显式传 `SamplingParams`
- 没解释输出结构
- 也没有提示这一步仍然是单卡路径

## 4. 本篇修改原则

### 4.1 保持单卡优先，不把 chunked prefill 一次塞进来

这一步先不追上游的：

- chunked prefill
- prefix-cache 驱动的 partial prefill
- worker 多进程通信

当前目标很明确：

> 先让你这份仓库的“单卡 generate 主循环”接口统一、状态清晰、测试可验收。

### 4.2 采样元数据必须从 `Sequence` 读，而不是从外面再拼一次

这一步要明确一个设计原则：

- `LLMEngine.add_request()` 把用户输入封装成 `Sequence`
- 从此以后，运行时只读 `Sequence`
- 不允许 `ModelRunner` 再回头依赖外部原始 `SamplingParams`

### 4.3 现在就为 Day6 留出 `run_model()` 边界

后面做 CUDA Graph 时，最自然的切入点就是：

- `run()` 负责准备输入、调用 `run_model()`、调用 sampler
- `run_model()` 负责“主干 hidden states -> logits”

所以本篇先把这层边界搭起来。

## 5. 逐步修改

## 5.1 先在 `ModelRunner` 里拆出采样元数据准备函数

修改位置：

- 文件：`nano_vll_repro/engine/model_runner.py`
- 锚点：定位到 `def prepare_decode(...)` 方法结束后、`@torch.inference_mode def run(...)` 之前
- 操作：在这两个方法之间插入下面这份完整方法定义

这一步目的很单纯：

- 不再在 `run()` 里现写温度张量
- 把 `temperature / top_k / top_p` 的来源固定为 `Sequence`

建议新增一个方法，放在 `prepare_decode()` 后、`run()` 前：

```python
def prepare_sampling_tensors(
    self,
    sequences: list[Sequence],
) -> tuple[torch.Tensor, torch.Tensor, torch.Tensor]:
    """
    输入：当前 step 要执行的序列列表。
    输出：
    1. temperatures: [batch]
    2. top_ks: [batch]
    3. top_ps: [batch]

    为什么要单独拆函数：
    - 采样元数据和 input_ids / positions 属于两种不同职责；
    - Day6 做 CUDA Graph 时，采样仍然在 graph 外执行，这里单独拆出来会更清楚。
    """
    temperatures = torch.tensor(
        [seq.temperature for seq in sequences],
        dtype=torch.float32,
        device=self.device,
    )
    top_ks = torch.tensor(
        [seq.top_k for seq in sequences],
        dtype=torch.long,
        device=self.device,
    )
    top_ps = torch.tensor(
        [seq.top_p for seq in sequences],
        dtype=torch.float32,
        device=self.device,
    )
    return temperatures, top_ks, top_ps
```

这里有一个经常被忽略的设计收益：

- 以后你要支持 per-sequence beam size、repetition penalty，也都可以照这个模式往下扩

## 5.2 把 `run_model()` 作为新的“主干前向 + logits 计算”边界

修改位置：

- 文件：`nano_vll_repro/engine/model_runner.py`
- 锚点：定位到刚刚插入的 `prepare_sampling_tensors(...)` 之后、`run(...)` 之前
- 操作：插入下面这份完整 `run_model(...)` 方法

当前 `run()` 直接做：

```python
logits = self.model(input_ids, positions)
```

这已经和 `02` 的模型接口不一致了。

你应该新增：

```python
@torch.inference_mode()
def run_model(
    self,
    input_ids: torch.Tensor,
    positions: torch.Tensor,
) -> torch.Tensor:
    """
    输入：
    - input_ids: 当前 step 的 token 输入
    - positions: 当前 step 的位置索引

    输出：logits

    这里故意把“主干 hidden states”和“lm_head logits”合成一个对外边界，
    原因是：
    - 单卡路径下调用方只关心 logits；
    - 但内部已经为 Day6 保留了把主干部分换成 CUDA Graph replay 的空间。
    """
    hidden_states = self.model(input_ids, positions)
    logits = self.model.compute_logits(hidden_states)
    return logits
```

然后在 `run()` 里，把旧的 `self.model(...)` 改成调用 `self.run_model(...)`。

## 5.3 同步把模型加载和 dtype 选择改成走 `Config`

修改位置：

- 文件：`nano_vll_repro/engine/model_runner.py`
- 锚点 1：定位到 `_load_model()` 里 `model = model.to(...)` 这一行，替换为下面这行
- 锚点 2：定位到 `allocate_kv_cache()` 里 `dtype=torch.float16` 这一行，替换为下面这行

当前 `_load_model()` 里直接：

```python
model = model.to(self.device, dtype=torch.bfloat16)
```

这必须改成：

```python
model = model.to(self.device, dtype=self.config.torch_dtype)
```

`allocate_kv_cache()` 里也不要再把 KV dtype 写死成 `torch.float16`，而是改成：

```python
dtype=self.config.kv_torch_dtype
```

原因不是“写得更优雅”，而是：

- Day5 单卡就可能在 CPU / CUDA / BF16 / FP16 间切换
- Day6 会更依赖主干 dtype 和 cache dtype 的分离能力

## 5.4 更新 `run()`：先准备输入，再跑模型，再按最后 token 采样

修改位置：

- 文件：`nano_vll_repro/engine/model_runner.py`
- 锚点：定位到 `@torch.inference_mode def run(...)`，从方法签名开始到 `return next_tokens.tolist()` 结束，整段替换为下面这份实现

这一步建议把 `run()` 收口成下面这种结构：

```python
@torch.inference_mode()
def run(self, sequences: list[Sequence], is_prefill: bool) -> list[int]:
    if not sequences:
        return []

    if is_prefill:
        input_ids, positions = self.prepare_prefill(sequences)
    else:
        input_ids, positions = self.prepare_decode(sequences)

    temperatures, top_ks, top_ps = self.prepare_sampling_tensors(sequences)
    logits = self.run_model(input_ids, positions)

    if is_prefill:
        # prefill 会把所有 prompt token 一次前向出来；
        # 这里只有每条序列最后一个 token 的 logits 才对应“下一个 token 的采样依据”。
        context = get_context()
        last_token_indices = context.cu_seqlens_q[1:] - 1
        logits = logits[last_token_indices.long()]

    next_tokens = self.sampler(logits, temperatures, top_ks, top_ps)

    # 一定要在 run 末尾清理 Context；
    # 否则后续 step 会读到上一个 batch 的元数据，Day6 做 graph 时更容易踩这个坑。
    reset_context()
    return next_tokens.tolist()
```

这一步你要特别注意一个顺序：

1. 先根据 `is_prefill` 选择输入准备逻辑
2. 再统一构造采样元数据
3. 再统一采样

不要把 prefill / decode 各自写一套采样逻辑，不然后面 graph fallback 会更难维护。

## 5.5 在 `Scheduler.postprocess()` 里补最基本的 cached-token 账本

修改位置：

- 文件：`nano_vll_repro/engine/scheduler.py`
- 锚点：定位到 `def postprocess(...)` 里 `seq.append_token(token_id)` 这一行之前
- 操作：先插入下面这行，再保留后面的 append 逻辑

当前 `postprocess()` 的 append 和 finish 逻辑大体没问题，但缺了一步非常重要的状态更新：

> 当前 step 被模型真正计算过的 token 数，应该在 append 新 token 之前记到账上。

推荐在 `append_token(token_id)` 之前加一行：

```python
# 这一轮进入模型计算的是“append 前的最后一个 token”；
# 所以在追加新 token 之前，旧长度对应的 token 已经进入 KV Cache，可以记为 cached。
seq.num_cached_tokens = seq.num_tokens
seq.append_token(token_id)
```

为什么不是 append 之后再 `+1`：

- 因为刚 append 进去的新 token 还没有被本轮模型处理
- 它会在下一轮 decode 时才真正写入 KV Cache

这条账本规则虽然简单，但后面：

- 统计 prefill token 数
- 判断 decode 上下文长度
- 处理 preemption

都会用到。

## 5.6 在 `LLMEngine.step()` 里改掉 prefill token 统计

修改位置：

- 文件：`nano_vll_repro/engine/llm_engine.py`
- 锚点：定位到 `def step(...)` 里原有的 “计算token数” 这一段
- 操作：把旧的 `if is_prefill: num_tokens = sum(len(seq) for seq in seqs)` 整段替换为下面这段

当前代码是：

```python
if is_prefill:
    num_tokens = sum(len(seq) for seq in seqs)
```

这不准确，因为：

- `len(seq)` 里可能已经包含上一步生成的 completion token
- 也包含当前 step 还没进入模型的新 token

更合理的统计方法是，在 `postprocess()` 之前按“本轮真正要计算的新 token 数”来记：

```python
if is_prefill:
    num_tokens = sum(max(seq.num_tokens - seq.num_cached_tokens, 1) for seq in seqs)
else:
    num_tokens = -len(seqs)
```

然后再调用：

```python
token_ids = self.model_runner.run(seqs, is_prefill)
finished_seqs = self.scheduler.postprocess(seqs, token_ids)
```

这样 tqdm 里的 prefill 吞吐量就不再是“混着历史 token 的伪统计”。

"""Day 5 测试脚本 - 单卡推理主链路验收

这一份测试文件专门验证 Day5 收口后的三件事情：

1. ModelRunner 能否把每条 Sequence 的采样参数提取成张量。

2. 第三个 smoke test 依赖本地模型权重与 CUDA；如果环境不满足，会显式跳过。

import os

import torch


# 让测试文件可以直接从仓库根目录旁路导入项目模块。

# 1. Sequence / SamplingParams：请求状态与采样配置
# 2. BlockManager / Scheduler：调度与 KV Cache 块管理

@torch.inference_mode()
def test_prepare_sampling_tensors() -> None:
    """测试 `ModelRunner.prepare_sampling_tensors()`。

    输入：
    1. 两条带不同 `temperature / top_k / top_p` 的 Sequence。
    2. 一个通过 `__new__` 绕过重量级初始化的 ModelRunner。

    输出：
    - 三个采样张量，形状都应为 `[batch]`。

    为什么这里不用真正初始化 ModelRunner：
    - 这个测试只关心“采样参数抽取逻辑”；
    - 真正初始化会加载模型、Tokenizer、KV Cache，成本太高；
    - 用最小依赖测试单一职责，回归更稳定。
    """

    # prepare_sampling_tensors 只依赖 self.device，因此不需要完整初始化。
    runner = ModelRunner.__new__(ModelRunner)
    runner.device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

    # 第一条序列走 greedy。
    seq1 = Sequence(
        [1, 2, 3],
        SamplingParams(
            temperature=0.0,
            top_k=0,
            top_p=1.0,
            max_tokens=8,
        ),
    )

    # 第二条序列走更常见的随机采样配置。
    seq2 = Sequence(
        [4, 5, 6],
        SamplingParams(
            temperature=0.8,
            top_k=20,
            top_p=0.9,
            max_tokens=16,
        ),
    )

    temperatures, top_ks, top_ps = ModelRunner.prepare_sampling_tensors(runner, [seq1, seq2])

    # 先断言张量形状，确认 batch 维度正确。
    assert temperatures.shape == (2,)
    assert top_ks.shape == (2,)
    assert top_ps.shape == (2,)

    # 再断言具体数值，确认每条序列各自的采样参数没有串位。
    assert temperatures.tolist() == [0.0, 0.8]
    assert top_ks.tolist() == [0, 20]
    assert torch.allclose(top_ps.cpu(), torch.tensor([1.0, 0.9]))

    print("✅ prepare_sampling_tensors 测试通过")


@torch.inference_mode()
def test_scheduler_postprocess_updates_cached_tokens() -> None:
    """测试 `Scheduler.postprocess()` 是否正确维护缓存账本。

    这个测试只验证一个很关键、也很容易写错的点：
    - 新 token 追加之前，旧长度对应的 token 应该记为 cached。

    如果这里记错，后面 decode 长度、吞吐量统计、甚至 preemption 都会出现连锁问题。
    """

    # 用一个最小配置对象满足 Scheduler 初始化要求。
    # 这里不需要真的创建完整 Config，因为 postprocess 只依赖这三个字段。
    config = SimpleNamespace(
        max_num_seqs=4,
        max_num_batched_tokens=64,
        eos=99999,
    )

    # BlockManager 也用最小可运行配置即可。
    block_manager = BlockManager(num_blocks=8, block_size=Sequence.block_size)
    scheduler = Scheduler(config, block_manager)

    # 构造一条正在 decode 的序列。
    seq = Sequence(
        [10, 11, 12],
        SamplingParams(
            temperature=0.7,
            top_k=10,
            top_p=0.9,
            max_tokens=4,
        ),
    )

    # 模拟这条序列已经进入 running 队列，并且已分配 block。
    seq.status = SequenceStatus.RUNNING
    block_manager.allocate(seq)
    scheduler.running.append(seq)

    # 记录 append 前的长度。
    old_num_tokens = seq.num_tokens

    # 生成一个普通 token，不触发 EOS，也不触发 max_tokens 完成。
    finished = scheduler.postprocess([seq], [13])

    # 这一步不应结束序列。
    assert finished == []
    assert seq.status == SequenceStatus.RUNNING

    # 新 token 已经 append 进去，所以总长度加一。
    assert seq.num_tokens == old_num_tokens + 1

    # cached token 数应等于“append 前的旧长度”。
    # 这是 Day5 账本逻辑的关键断言。
    assert seq.num_cached_tokens == old_num_tokens

    print("✅ scheduler.postprocess 账本测试通过")

## 7. 常见错误

### 7.1 `run()` 里忘记调用 `compute_logits()`

后果：

- 采样器拿到的是 hidden states，不是 vocab logits
- 报错可能不是第一时间出现，而是 shape 在 softmax 时才错

### 7.2 `reset_context()` 忘记放在 `run()` 末尾

后果：

- 当前单卡路径偶尔还能侥幸通过
- 到 Day6 做 graph replay 时会非常难查

### 7.3 `num_cached_tokens` 在 append 之后才更新

后果：

- 会把“刚生成但尚未进入模型的新 token”也记成 cached
- decode 上下文长度会慢慢错位

### 7.4 `LLMEngine.step()` 继续拿 `len(seq)` 统计 prefill token

后果：

- 进度条里的吞吐量严重失真
- Day7 benchmark 时你会得到看起来很“漂亮”但没有意义的数据
